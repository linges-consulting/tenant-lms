## 3. Findings

### Finding 1 — `chapter.sequence_order` is module-scoped in seed data but training-scoped in the backend's gating check

**Severity:** medium · **Origin:** pre-existing in seed + backend, not introduced by the change_list review

**Repro**
1. Log in as `cp-dual@lms.com`.
2. Open the seeded `Test3` training (`/manage/learn/39a925a3-92a6-4516-8a29-8d5beb8d0913`).
3. Module 1 has chapters `qweqwe` (sequence_order=1, completed) and `12312312` (sequence_order=2, active).
4. Module 2 has a single chapter `retertert` with `sequence_order=1`.
5. Click "Next Lesson" on the active `12312312` chapter.
6. **Observed:** alert "Failed to mark chapter as complete. You may need to complete previous chapters first." with backend 403.

**Root cause sketch** — `complete_chapter` looks up the previous chapter via:
```python
select(Chapter)
  .where(Chapter.training_id == training_id)
  .where(Chapter.sequence_order < chapter.sequence_order)
  .order_by(Chapter.sequence_order.desc())
  .limit(1)
```
For `12312312` (seq=2) this matches two rows with seq=1 — `qweqwe` (completed) and `retertert` (not completed). With no module filter or stable secondary sort, the DB returns whichever it likes, and if it picks `retertert` the 403 fires.

**Recommended next step**
- **Quick fix:** add `.where(Chapter.module_id == chapter.module_id)` (or the orphan equivalent) so the lookup is module-scoped — matches how the sidebar lock UI already groups by module.
- **Or:** make sequence_order globally unique within a training at write time, and migrate the seed.
- Backstop with a backend test that creates two modules with overlapping per-module seq numbers and proves the previous-chapter check picks deterministically.

### Finding 2 — `total_chapters` mismatch between `/trainings/{id}` and `/trainings/{id}/structure`

**Severity:** low · **Worth a quick look**

`GET /api/v1/trainings/39a925a3-…` reported `total_chapters: 5` while `GET /api/v1/trainings/39a925a3-…/structure` only returned 3 chapters across the two modules (no orphan chapters). Likely a soft-deleted-row count or an older module's chapters still being counted by the `count()` query. Not user-visible until they hit the completion screen and the "Chapters Completed: 1 / 5" feels wrong.
